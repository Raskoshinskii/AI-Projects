### Прогнозирование Оттока Телеком Оператора
### Описание Проекта 
Данные для текущего проекта были получены в результате конкурса, проведенного французской телекоммуникационной компанией Orange. Исходный датасет содержал персональную информацию клиентов, поэтому данные предварительно анонимизированы: из набора данных была удалена вся личная информация, позволяющая идентифицировать пользователей, а названия и описания признаков, предназначенных для составления прогнозов, не были предоставлены.

### Цель Проекта
Научиться находить пользователей, склонных к оттоку. Если заранее научиться находить таких пользователей с достаточной точностью, можно эффективно управлять оттоком:
  - Определите причины оттока
  - Помогать пользователям из группы риска решить их проблемы и задачи
  - Проводить кампании по удержанию пользователей склонных к оттоку

С точки зрения машинного обучения, мы должны создать модель прогнозирования - модель, которая предсказывает вероятность того, что пользователь покинет сервис. В классической постановке строится вероятностная модель бинарной классификации, где целевой класс - это пользователи, склонные к оттоку. Вероятность принадлежности пользователя к целевому классу - вероятность оттока (целевой признак). Соответственно, чем больше эта вероятность, тем больше шансов, что пользователь откажется от использования нашего сервиса.

### Задачи Проекта
Для достижения цели были сформулированы следующие задачи:
  - Первоначальный Анализ Данных: особенности, размер, определение закономерностей и структуры данных
  - Отбор Признаков: определение полезных признаков, позволяющие правильно классифицировать пользователей, которые «уйдут» или «останутся»
  - Обработка Данных: определение наилучших ( imputers, categorical encoders) и методов для борьбы с несбалансированными классами (Random Over/Undersampling, SMOTE)
  - Выбор Модели: определить модель, которая наилучшим образом соответствует заявленной цели проекта
  - Оптимизация Модели: определение оптимальных гиперпараметров модели, обеспечивающих наилучшее качество классификации

### Метрики
В данном проекте нет необходимости самостоятельно определять метрику, так как она уже определена соревнованием: `ROC-AUC` Однако мы также будем использовать вспомогательные метрики, такие как `Precision` и `Recall`, для отслеживания числа `FalsePositive` и `FalseNegative`, а также `F1-Score`

Все метрики были рассчитаны с использованием `StratifiedCrossValidation (n_folds = 5)`

### Pipeline Проекта 
Для обеспечения быстрой и надежной (т.е. без утечки данных) обработки признаков был определен следующий `Pipeline`

![Untitled Diagram (7)](https://user-images.githubusercontent.com/56967765/136270308-c3e7faf0-a31e-4ea2-9982-e41757fdedbf.png)

Пропущенные значения для числовых и категориальных признаков были заполнены используя `SimpleImputer`:
  - Категориальные Признаки: `SimpleImputer (strategy = 'constant', fill_value = 'unknown')`
  - Числовые Признаки: `SimpleImputer (strategy = 'mean')`

Кодирования и масштабирования признаков:
  - Кодирование Признаков: `TargetEncoder()`
  - Масштабирование Признаков: `StandardScaler()`

Все элементы предварительной обработки данных были проверены на перекрестной проверке, и были выбраны лучшие (т.е. обеспечивающие лучший показатель `ROC-AUC`). Итоговый `Pipeline` принимает финальную выборку (после отбора признаков).

### Отбор Признаков
Следует сказать, что в наборе данных было много пропущенных значений. Первым шагом было выявление признаков, которые полностью состояли из пропущенных значений, такие признаки были исключены из рассмотрения. Кроме того, была построена гистограмма для визуализации соотношения пропущенных значений для каждого признака.

![Screenshot 2021-10-06 063123](https://user-images.githubusercontent.com/56967765/136271093-402aa22f-eb95-4a59-9298-5c5449e6f33c.png)

Было решено исключить признаки с коэффициентом NaN более 90%, поскольку мы вряд ли сможем восстановить эти зависимости. Это позволило исключить довольно большое количество числовых признаков и несколько категориальных. Отобранные таким способом признаки попали в дальнейший анализ.

Затем категориальные и числовые признаки были проанализированы отдельно, чтобы определить связь с целевой функцией:
  - Категориальные Признаки: `Значение V-Краммера`
  - Числовые Признаки: `Значение хи-квадрат`

Рассчитанные значения использовались для определения признаков наиболее сильно коррелирующих с целевой переменной. Поскольку их было немного, было решено итеративно добавлять данные признаки в модель, строить её на текущем наборе признаков и оценивать качество. Были получены следующие результаты.

### Методы Борьбы с Несбалансированными Классами
Набор данных несбалансирован. Соотношение классов: `{-1.0: 0.924746, 1.0: 0.075254}`

Были использованы следующие методы `Undersampling`:
  - `Random Undersampling`
  - `One-Sided Selection`
  - `Under-Sampling with Cluster Centroids`
  - `Nearmiss 1 & 2 & 3`

Были использованы следующие методы `Oversampling`:
  - `Random Oversampling`
  - `SMOTE (Synthetic Minority Oversampling Technique)` 
  - `ADASYN (Adaptive Synthetic Sampling Approach)`
  - `Combination SMOTE + Tomek Links`

Эти методы не смогли кардинально улучшить ситуацию. В качестве основных были выбраны три кандидата:
  - Присвоение весов наблюдениям (`F1-Score: 0.282003`, `ROC-AUC: 0.733730`)
  - Random Oversampling (`F1-Score: 0.647494`, `ROC-AUC: 0.774711`)
  - Random Undersampling (`F1-Score: 0.284221`, `ROC-AUC: 0.733171`)

`Random Oversampling` не использовалась в окончательном решении, так как существовал риск переобучения из-за дублирования объектов. Окончательно, был применен метод: присвоение весов наблюдениям  (более надежный).

### Оптимизация Гиперпараметров Модели 
Итоговая модель: `Gradient Boosting Classifier`

Выбор гиперпараметров модели производился с использованием **байесовского подхода** с использованием модуля `HyperOpt`. Количество параметров модели велико, поэтому использование стандартного `GridSearchCV` потребывало бы очень много времени. Подбор оптимальных гиперпараметров лишь немного улучшил качество финального решения (`ROC-AUC: 0.7364 (CV)`

Использование `CatBoost` и оптимизация гиперпараметров позволили занять (в настоящее время <a href='https://www.kaggle.com/c/telecom-clients-prediction2/leaderboard'> 7-е место </a> с качеством: `0.73128`)
